# Бэкенд для Интукод на Go + Gin + GORM

## Постановка задачи

Тебе предстоит разработать полноценный REST API‑сервер для школы Интукод.
Сервер будет хранить информацию о студентах, их группах, оплатах и заметках наставников.

Цель — получить удобный внутренний бэкенд, который позволяет:

- смотреть списки студентов и групп;
- фильтровать студентов по статусу обучения и оплатам;
- работать с заметками по каждому студенту;
- считать простую статистику по трудоустройству в группе.

Все данные должны храниться в базе данных (PostgreSQL) через GORM, а HTTP‑слой реализован на Gin.

## Модели и данные

Минимальный набор сущностей (можно добавить свои поля при желании):

- **Student**

  - `full_name` — ФИО студента;
  - `email` — почта;
  - `telegram` — ник или ссылка;
  - `group_id` — ссылка на группу (FK);
  - `tuition_total` — полная стоимость обучения (в условных единицах, `int`);
  - `tuition_paid` — сколько уже оплачено;
  - `payment_status` — статус оплаты: одно из значений:
    - `"paid"` — полная оплата;
    - `"unpaid"` — нет оплаты;
    - `"partial"` — неполная оплата;
  - `study_status` — статус обучения: одно из значений:
    - `"learning"` — в процессе обучения;
    - `"job_search"` — ищет работу;
    - `"offer"` — получил оффер;
    - `"working"` — вышел на работу.

- **Group**

  - `title` — название группы (например, `"Go весна (вечер)"`);
  - `current_week` — текущая неделя обучения (от `1` до `15`);
  - `total_weeks` — общее количество недель обучения (`15`);
  - `is_finished` — флаг, что группа закончила обучение.

- **Note**
  - `student_id` — ссылка на студента (FK);
  - `author` — кто оставил заметку (например, наставник);
  - `text` — текст заметки;

Требования к моделям:

- Используй `gorm.Model`
- Для всех экспортируемых полей добавь `json`‑теги в `snake_case`, например:
- Для структур, которые используются как тело запроса для POST и PATCH создавай отдельные структуры, чтобы не использовать гормовские.

## Эндпоинты и фильтры

Все ответы сервера — **JSON**.
При ошибках сервер должен возвращать объект вида:

```json
{ "error": "описание ошибки" }
```

и подходящий HTTP‑код (`400`, `404`, `500` и т.д.).

Для вставки кодов используй пакет `http` из `net/http`.

### Студенты

- **Вывод всех студентов**

  `GET /students` — вернуть список всех студентов.

- **Вывод всех студентов из определённой группы**

  `GET /students?group_id=2`

- **Вывод всех заметок об определённом студенте**

  `GET /students/:id/notes`

- **Вывод всех студентов, сделавших оплату**

  `GET /students?payment_status=paid`

- **Вывод всех студентов, не сделавших оплату**

  `GET /students?payment_status=unpaid`

- **Вывод всех студентов, сделавших неполную оплату**

  `GET /students?payment_status=partial`

- **Вывод студентов по одному из статусов обучения**

  Используй query‑параметр `study_status`:

  - `GET /students?study_status=learning` — в процессе обучения;
  - `GET /students?study_status=job_search` — ищет работу;
  - `GET /students?study_status=offer` — получил оффер;
  - `GET /students?study_status=working` — вышел на работу.

### Группы

- **Вывод всех групп**

  `GET /groups`

- **Вывод всех групп, находящихся на определённой неделе (1–15)**

  `GET /groups?week=3`

  Сервер должен вернуть только те группы, у которых `current_week == 3`.

- **Вывод групп, окончивших обучение**

  `GET /groups?finished=true`

  Критерий окончания можно выбрать один:

  - либо `is_finished == true`,
  - либо `current_week >= total_weeks`.

- **Получить процент получивших оффер из определённой группы**

  `GET /groups/:id/stats/offer`

  В ответ нужно вернуть JSON с процентом студентов в этой группе, у которых `study_status == "offer"`.  
  Например:

  ```json
  {
    "group_id": 2,
    "offer_percent": 37.5
  }
  ```

### CRUD для всех моделей

Для каждой из моделей (`Student`, `Group`, `Note`) должны быть реализованы все CRUD‑операции:

- `POST /students` — создать студента;
- `GET /students/:id` — получить студента по ID;
- `PATCH /students/:id` — частично обновить данные студента;
- `DELETE /students/:id` — удалить студента;

и аналогичные маршруты для:

- `/groups` — работа с группами;
- `/notes` — работа с заметками (в том числе `GET /notes/:id`, `POST /notes`, `PATCH /notes/:id`, `DELETE /notes/:id`).

При создании/обновлении студента нужно валидировать поля:

- `full_name` — непустая строка;
- `payment_status` — одно из `paid|unpaid|partial`;
- `study_status` — одно из `learning|job_search|offer|working`;
- `group_id` — должен ссылаться на существующую группу.

## Releases

Это рекомендуемый план релизов, но не строго фиксированный.

### Release 0 — Проектирование

- Сначала **на бумаге или в отдельном файле** спроектируй:
  - структуры `Student`, `Group`, `Note`;
  - поля и их типы;
  - все необходимые эндпоинты (маршруты, методы, формат ответа и тела запроса).
- Согласуй структуру данных и список роутов с наставником перед тем, как писать код.

### Release 1 — Каркас проекта и подключение зависимостей

- Создай новый Go‑проект и инициализируй модуль: `go mod init <module_path>`.
- Установи зависимости:
  - Gin (`github.com/gin-gonic/gin`);
  - GORM (`gorm.io/gorm`, `gorm.io/driver/postgres`).
- Подготовь базовый сервер Gin с тестовым роутом `GET /ping` → `{ "message": "pong" }`.
- Настрой подключение к PostgreSQL и проверь, что соединение устанавливается без ошибок.

### Release 2 — Модели и миграции (GORM)

- Опиши модели `Student`, `Group`, `Note` с `gorm.Model` и нужными полями.
- Настрой `AutoMigrate` для всех трёх моделей.
- После запуска приложения проверь в pgAdmin (или другом клиенте), что таблицы создались корректно.

### Release 3 — Базовый CRUD для групп и студентов

- Реализуй CRUD для `Group`:
  - `POST /groups`, `GET /groups`, `GET /groups/:id`, `PATCH /groups/:id`, `DELETE /groups/:id`.
- Реализуй CRUD для `Student`:
  - `POST /students`, `GET /students`, `GET /students/:id`, `PATCH /students/:id`, `DELETE /students/:id`.
- При создании студента:
  - проверяй, что `group_id` существует;
  - выставляй корректные значения `payment_status` и `study_status` (валидация).

### Release 4\* — Фильтры по группам, оплатам и статусам

Если этот релиз занимает много времени, то пропусти его и вернись к нему позже, после выполнения остальных.

- Добавь поддержку фильтров в `GET /students`:
  - `group_id` — выбор студентов конкретной группы;
  - `payment_status` — `paid|unpaid|partial`;
  - `study_status` — `learning|job_search|offer|working`.
- Добавь роут для выдачи студентов по группе:
  - `GET /groups/:id/students` — все студенты этой группы.
- Убедись, что сочетания фильтров работают корректно (например, `GET /students?group_id=2&payment_status=partial`).

### Release 5 — Заметки по студентам

- Реализуй модель и миграции для `Note` (если ещё не сделал).
- Подготовь роуты:
  - `GET /students/:id/notes` — все заметки по студенту;
  - `POST /notes` — создание заметки (в теле запроса должен быть `student_id`);
  - `PATCH /notes/:id` — редактирование текста;
  - `DELETE /notes/:id` — удаление заметки.
- Валидация:
  - `student_id` должен ссылаться на существующего студента;
  - текст заметки — непустая строка.

### Release 6 — Группы по неделям и статистика офферов

- В `GET /groups` добавь поддержку фильтров:
  - `week` — вернуть только группы с `current_week == week`;
  - `finished=true` — вернуть только завершённые группы.
- Реализуй `GET /groups/:id/stats/offer`:
  - посчитай общее количество студентов в группе;
  - посчитай количество студентов со статусом `offer`;
  - верни процент в поле `offer_percent` (тип `float64`).
- Продумай, что делать, если в группе пока нет студентов (например, вернуть `offer_percent = 0`).

## Критерий приёмки

Задание считается выполненным, если:

- сервер поднимается без ошибок и подключается к PostgreSQL;
- `AutoMigrate` создаёт таблицы для всех трёх моделей;
- реализованы CRUD‑эндпоинты для `Student`, `Group`, `Note`;
- работают все перечисленные фильтры по студентам (группа, оплата, статус обучения);
- работают фильтры по группам (неделя, завершённость);
- `GET /students/:id/notes` возвращает список заметок по студенту;
- `GET /groups/:id/stats/offer` возвращает корректный процент студентов с оффером;
- все ошибки возвращаются в виде JSON `{ "error": "..." }` с подходящими HTTP‑кодами.

После самостоятельной проверки протестируй все эндпоинты в Postman и только потом отправляй работу наставнику.
